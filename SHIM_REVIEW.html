📐 What Was Built

Full Shim Architecture (not Spike 1 Namespace Shadowing)

The WLA_Playwrite implementation takes a different route than the Spike 1 proof-of-concept. Instead of namespace shadowing + reflection, it builds a full Selenium.Playwright.Shim library that provides all OpenQA.Selenium.* and TRGR.Quality.QedArsenal.* types in-namespace. DriverFactory.cs is modified to create PlaywrightWebDriver directly. All other framework and test files compile against the shim instead of real Selenium/QedArsenal NuGets.

✅ What's Correct

  • By mechanism strings match ByConverter exactly
  • SyncHelper.RunSync uses Task.Run(...).GetResult() — no deadlock risk
  • WebDriverWait.Until() properly polls with exception handling
  • TRGR.Quality stubs delegate correctly to IWebDriver
  • DriverFactory.InitializeChromeDriver creates PlaywrightWebDriver
  • Navigation: GoToUrl, Back, Refresh properly implemented
  • Cookie management fully implemented
  • Screenshot via Page.ScreenshotAsync() — correct

❌ What's Broken

  • Browser opens but stays on about:blank — sign-on never navigates
  • Implicit wait is 0ms — elements fail to resolve on page load
  • JavascriptClick passes PlaywrightWebElement as JS arg — crashes at runtime
  • WaitForPageLoad / WaitForJavaScript missing from shim QedArsenal stubs

🐛 Root Causes — "Browser Opens, No Actions"

1 Implicit Wait Is Zero — Elements Time Out Instantly

PlaywrightTimeouts.ImplicitWait defaults to 0 seconds. Every call to FindElement passes Timeout = 0 to locator.WaitForAsync(). Playwright treats 0 ms as an immediate check — if the element is not attached at that exact millisecond, it throws PlaywrightException, which becomes NoSuchElementException. The outer WebDriverWait polls but each inner attempt never waits for the DOM to settle. Fast-loading elements may still resolve, but any element behind a network call or animation silently fails. 

// PlaywrightOptions.cs
internal class PlaywrightTimeouts : ITimeouts
{
    public TimeSpan ImplicitWait { get; set; } = TimeSpan.FromSeconds(0);  // ← BUG: 0ms
}

// PlaywrightWebDriver.cs — FindElement
var timeout = _options.GetImplicitWaitMs();  // returns 0.0f
SyncHelper.RunSync(() => locator.First.WaitForAsync(new LocatorWaitForOptions
{
    Timeout = timeout   // 0ms → Playwright checks once, immediately throws if not there
}));

Fix: Set ImplicitWait = TimeSpan.FromSeconds(10) as default. This gives Playwright a sensible retry window matching the original Selenium behavior.

2 JavascriptClick Crashes at Runtime — Wrong JS Arg Type

The shim's ClickExtensions.JavascriptClick() passes a PlaywrightWebElement to ExecuteScript("arguments[0].click()", element). PlaywrightWebDriver.ExecuteScript calls MapArguments, which calls pwe.Locator.ElementHandleAsync() to convert the locator to a Playwright IElementHandle. Playwright has deprecated ElementHandleAsync() and it can return null or throw when called on strict-mode locators. The wrapped WebDriverException propagates up. Framework code that wraps this in SafeMethodExecutor silently swallows it — the click never happens. 

// PlaywrightWebDriver.cs — MapArguments
if (args[i] is PlaywrightWebElement pwe)
{
    mapped[i] = SyncHelper.RunSync(() => pwe.Locator.ElementHandleAsync());
    // ↑ ElementHandleAsync is deprecated; returns null on strict locators
}

// ClickExtensions.cs
public static void JavascriptClick(this IWebDriver driver, IWebElement element)
{
    ((IJavaScriptExecutor)driver).ExecuteScript("arguments[0].click();", element);
    // ↑ element is PlaywrightWebElement — MapArguments fails to resolve it
}

Fix: Replace ElementHandleAsync() with page.EvaluateAsync("(el) => el.click()", locator) directly. Do not pass PlaywrightWebElement through the JS arg pipeline. Instead, detect it in ExecuteScript and rewrite the call entirely.

3 Missing WaitForPageLoad / WaitForJavaScript in QedArsenal Stubs

BaseWebObject's constructor calls DriverExtensions.WaitForPageLoad() and DriverExtensions.WaitForJavaScript() statically. These are provided by Framework.Common.UI's WebDriverExtensions.cs, which internally calls the QedArsenal extension methods on IWebDriver. The shim's WaitExtensions stub does not include WaitForPageLoad(this IWebDriver driver) or WaitForJavaScript(this IWebDriver driver). If the Framework facade calls those methods via the QedArsenal extension interface, they resolve to missing stubs — compile error or silent no-op depending on how they're referenced. Every page object constructor silently skips the DOM-ready wait. 

// BaseWebObject.cs — constructor (Framework.Common.UI, UNCHANGED)
public BaseWebObject()
{
    DriverExtensions.WaitForPageLoad();     // ← calls QedArsenal WaitForPageLoad
    DriverExtensions.WaitForJavaScript();   // ← calls QedArsenal WaitForJavaScript
}

// Shim WaitExtensions.cs — MISSING these two methods:
// WaitForPageLoad(this IWebDriver driver) — NOT PRESENT
// WaitForJavaScript(this IWebDriver driver) — NOT PRESENT

Fix: Add both methods to the shim's TRGR.Quality\Extensions\WaitExtensions.cs. Implement WaitForPageLoad using page.WaitForLoadStateAsync(LoadState.DOMContentLoaded) and WaitForJavaScript using page.WaitForFunctionAsync("() => document.readyState === 'complete'").

4 GotoAsync Uses Url Setter But Sign-On Calls Navigate().GoToUrl()

Two navigation paths exist. PlaywrightWebDriver.Url setter calls Page.GotoAsync(value) with no explicit wait-until state — it uses Playwright's default (Load event). PlaywrightNavigation.GoToUrl uses DOMContentLoaded with 60s timeout. The WLA sign-on page is a JavaScript-heavy SPA. When the framework uses the Url setter (common in BrowserPool sign-on flow), navigation completes at the network level but the React app hasn't rendered. Subsequent FindElement calls execute against an empty DOM shell. All locators return NoSuchElementException. Sign-on fails silently (caught by SafeMethodExecutor). Browser stays at the sign-on URL but no actions are performed. 

// PlaywrightWebDriver.cs
public string Url
{
    set
    {
        SyncHelper.RunSync(() => Page.GotoAsync(value));
        // ↑ No WaitUntil specified — uses Playwright default (Load event)
        // SPA may not be render-ready after Load fires
    }
}

// PlaywrightNavigation.cs — correctly uses DOMContentLoaded + 60s
SyncHelper.RunSync(() => _driver.Page.GotoAsync(url, new PageGotoOptions {
    WaitUntil = WaitUntilState.DOMContentLoaded, Timeout = 60000 }));

Fix: Update the Url setter to use the same options as GoToUrl: WaitUntilState.DOMContentLoaded with a 60s timeout. After navigation, add a Page.WaitForLoadStateAsync(LoadState.NetworkIdle) call to ensure the React shell has settled before element lookups begin.

⚠️ Secondary Risks (After Primary Fixes)

⚠️ WaitForAsync(Timeout=0) Semantics

Even after fixing the default, if code explicitly calls driver.Manage().Timeouts().ImplicitWait = TimeSpan.Zero (via DisableTimeout()), FindElement will again pass Timeout=0 to Playwright. Playwright treats 0 as "use browser default" (30s) in some builds and "instant fail" in others. Verify behavior with the specific Playwright v1.44.0 build in use.

⚠️ SendKeys Special Keys Only PressAsync

PlaywrightWebElement.SendKeys routes special-key sequences (Unicode \ue000–\ue03d) to Locator.PressAsync(mapped). But Keys.Tab, Keys.Enter etc. from the framework may arrive mixed with regular text (e.g., "value" + Keys.Tab). The current code only handles pure-special or pure-text — mixed input breaks.

⚠️ WaitForJavaScript on SPA Pages

WLA uses a React SPA. document.readyState === 'complete' fires before React finishes rendering. The WaitForJavaScript stub should additionally wait for any window.jQuery or app-specific ready signal if the original QedArsenal implementation did so.

⚠️ Frame Switching State

PlaywrightTargetLocator.Frame() sets CurrentFrame on the driver. This is a driver-level field, not page-level. If a test switches tabs then back, CurrentFrame may still point to a frame from the previous page. Add CurrentFrame = null in SetActivePage() (already present) but also on any navigation in Url setter and GoToUrl.

⚠️ GetParentElement JS Return Type

ElementRetrievalExtensions.GetParentElement does: (IWebElement)driver.ExecuteScript("return arguments[0].parentNode;", element). PlaywrightWebDriver.ExecuteScript returns object from Page.EvaluateAsync<object> — a JSON-deserialized value, not an IWebElement. The cast to IWebElement throws InvalidCastException at runtime.

⚠️ FindPublishDirectory Filesystem Scan

The static constructor in PlaywrightWebDriver scans C:\Github\*\src\Shim\publish looking for the Playwright driver. This will fail on CI machines with different drive letters or non-standard repo paths. Hardcoded drive and directory names (C, Github, git, repos) make this brittle. Use an environment variable or embedded resource instead.

Verdict on Approach Choice

The full shim approach is architecturally superior for the full migration goal. It eliminates QedArsenal entirely, creates a reusable shim library for other products, and avoids fragile reflection. The Spike 1 approach was valid for a 5-day feasibility proof but is not the right foundation for production. Continue with the shim — fix the 4 root causes below.

🔧 Fix Plan — Priority Order

1

Set default ImplicitWait to 10 seconds

In PlaywrightTimeouts, change TimeSpan.FromSeconds(0) to TimeSpan.FromSeconds(10). This gives every FindElement call a 10-second Playwright auto-wait window, matching the original Selenium implicit wait behavior. Estimated fix: 1 line.

2

Fix Url setter navigation

In PlaywrightWebDriver.Url setter, add PageGotoOptions { WaitUntil = WaitUntilState.DOMContentLoaded, Timeout = 60000 } — match PlaywrightNavigation.GoToUrl exactly. Then add Page.WaitForLoadStateAsync(LoadState.NetworkIdle, timeout: 10000) wrapped in try-catch (NetworkIdle may not fire on all pages). Estimated fix: 5 lines.

3

Add WaitForPageLoad and WaitForJavaScript to shim

In TRGR.Quality\Extensions\WaitExtensions.cs, add: 

public static void WaitForPageLoad(this IWebDriver driver)
{
    if (driver is Selenium.Playwright.Shim.Impl.PlaywrightWebDriver pw)
        SyncHelper.RunSync(() => pw.Page.WaitForLoadStateAsync(
            Microsoft.Playwright.LoadState.DOMContentLoaded));
}

public static void WaitForJavaScript(this IWebDriver driver)
{
    if (driver is Selenium.Playwright.Shim.Impl.PlaywrightWebDriver pw)
        SyncHelper.RunSync(() => pw.Page.WaitForFunctionAsync(
            "() => document.readyState === 'complete'"));
}
4

Fix JavascriptClick — bypass ElementHandleAsync

In PlaywrightWebDriver.ExecuteScript, detect when script is "arguments[0].click();" and the first arg is a PlaywrightWebElement, then route directly to pwe.Locator.ClickAsync() instead of going through EvaluateAsync. Alternatively, remove ElementHandleAsync entirely from MapArguments and pass a JS handle via Locator.EvaluateAsync("el => el", []) which returns a serializable ref. Estimated fix: 15 lines.
