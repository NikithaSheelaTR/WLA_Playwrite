Selenium to Playwright Migration

Drop-in Shim Architecture & Approach — WestlawAdvantage Test Automation

Approach

We created a drop-in shim assembly (Selenium.Playwright.Shim.dll) that re-declares all OpenQA.Selenium namespaces, interfaces, and classes — with each implementation internally delegating to Microsoft Playwright — so that existing test code, page objects, and base classes compile and run unchanged, with the only modification being swapping the Selenium NuGet reference for the shim DLL and routing DriverFactory to instantiate PlaywrightWebDriver instead of ChromeDriver.

Key Constraint: Zero changes to existing test files, base classes, page objects, or utility classes. Only DriverFactory.cs and the new shim assembly were modified/created.

Before vs After

BEFORE — Selenium

  • Selenium.WebDriver NuGet package
  • chromedriver.exe binary required
  • ChromeDriver communicates via WebDriver protocol (JSON Wire / W3C)
  • Separate process for browser driver
  • Version coupling: ChromeDriver must match Chrome version

AFTER — Playwright Shim

  • Selenium.Playwright.Shim.dll (single assembly)
  • No chromedriver.exe needed
  • Playwright communicates via Chrome DevTools Protocol (CDP)
  • Built-in browser management via .playwright/node/
  • Uses system Chrome (channel: "chrome") — no version coupling

Key Implementation Details

1. Async-to-Sync Bridge (SyncHelper)

Playwright is async-only. The existing test framework is synchronous. SyncHelper runs each async call on a dedicated Thread with a null SynchronizationContext to prevent deadlocks under vstest.

// Runs async Playwright calls synchronously on a clean thread
public static T RunSync<T>(Func<Task<T>> asyncFunc)
{
    T result = default;
    Exception caught = null;
    var thread = new Thread(() => {
        SynchronizationContext.SetSynchronizationContext(null);
        try { result = asyncFunc().GetAwaiter().GetResult(); }
        catch (Exception ex) { caught = ex; }
    });
    thread.Start();
    thread.Join();
    if (caught != null) ExceptionDispatchInfo.Capture(caught).Throw();
    return result;
}

2. Browser Maximize (CDP-based)

Selenium's --start-maximized + Window.Maximize() is replicated using Chrome DevTools Protocol to get the actual window and set windowState: "maximized".

// CDP approach — matches what Selenium's --start-maximized does
var cdp = await page.Context.NewCDPSessionAsync(page);
var window = await cdp.SendAsync("Browser.getWindowForTarget");
await cdp.SendAsync("Browser.setWindowBounds", new {
    windowId = window.Value.GetProperty("windowId").GetInt32(),
    bounds = new { windowState = "maximized" }
});

3. ExecuteScript Translation

Selenium's IJavaScriptExecutor.ExecuteScript() patterns are mapped to Playwright equivalents:

Selenium PatternPlaywright Translation
return document.readyState; Page.EvaluateAsync("() => { return document.readyState; }")
arguments[0].click() Locator.EvaluateAsync("el => el.click()")
arguments[0].focus() Locator.FocusAsync()
return arguments[0].scrollHeight; Locator.EvaluateAsync("(el) => { return el.scrollHeight; }")

4. SendKeys Optimization

Single-character input uses a fast path via Page.Keyboard.TypeAsync() instead of element-level PressSequentiallyAsync(), reducing per-character time from 2–5 seconds to ~20ms on dynamic pages (like CIAM sign-on).

5. FindElement Timeout Handling

When timeout is 0 (immediate check), the shim uses Locator.CountAsync() instead of WaitForAsync() to avoid Playwright's default 30-second wait.

6. Assembly Resolution

A static constructor registers an AppDomain.AssemblyResolve handler that searches the shim directory, publish folder, and PDB-embedded source paths to load BCL dependencies even when vstest copies DLLs to a deployment directory.
